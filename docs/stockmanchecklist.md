# Stockman 交互回归清单

> **注意**：此清单对应的是旧版 Qt6 C++ Stockman 客户端（覆盖 Shell / Tasking / Loot / Chat / Topology / Transfer / SOCKS / Proxy / SSH 等完整运维路径）。在 `academic` 分支上 Stockman 已经重写为 Wails v2 + Vue 3 的轻量演示客户端，**不实现本文列出的大多数交互路径**。本文保留作为历史回归参考，新版客户端的验证方式请见 `clientui/README.md` 与 `docs/stockmansetup.md`。新版演示路径要点可以简单总结为：连接 → 拓扑切换（力导向/树状）→ 选中节点 → 控制台发 DTN → 事件时间线出现对应事件 → 用 Sleep 改 duty-cycle 再发 DTN → PruneOffline 回正视图。

本文用于固定 Stockman 在毕业设计阶段最容易出现回归的交互路径，重点回答两个问题：

- 这次界面重构之后，核心操作是否仍然稳定。
- 答辩或演示时，如果老师问“你怎么验证 UI 没有乱”，可以直接据此说明。

适用范围：

- 旧版 Qt6 C++ `clientui/` 页面结构重构后
- Shell / Tasking / Loot / Chat / Topology 相关交互修改后
- 提交前的手工回归

---

## 1. 执行原则

- 每次回归至少连接一个可正常响应的 Kelpie/Flock 环境。
- 如果修改涉及节点切换、流、下载上传、SOCKS、拓扑或列表刷新，必须逐项走完对应清单。
- 回归时尽量保留一份屏幕录像或操作日志，便于答辩时说明“我不仅写了代码，也验证了交互稳定性”。

---

## 2. 必测主路径

### 2.1 节点切换不会误杀 Shell

步骤：

1. 连接 Kelpie，等待节点树出现至少两个节点。
2. 在一个节点上打开 Shell。
3. 在 Shell 中执行一条有明显输出的命令，例如 `pwd` 或 `whoami`。
4. 切换到另一个节点，再切回原节点。
5. 等待后台 snapshot 刷新至少两轮。

预期：

- Shell 只会在真实切换到其他节点时关闭。
- 在“同一节点被后台刷新后重新选中”的情况下，Shell 不会被 UI 自己断开。
- 不会出现输入框突然禁用、输出框被清空、状态标签错误回到 `disconnected` 的现象。

### 2.2 节点切换不会误杀 SSH

步骤：

1. 在 `Tasking -> SSH` 中建立一个 SSH session 或 SSH tunnel。
2. 保持 `Tasking` 页停留在 SSH 标签页。
3. 切换节点，再切回原节点。
4. 等待一次 snapshot 刷新和一次 `stream_event` 刷新。

预期：

- 当前 SSH 流不会因为 UI 页面刷新而从列表中消失。
- 若真实切换到其他节点，列表内容应随目标节点变化；若只是同一节点刷新，条目应保持稳定。
- 不会出现迟到的 `Closed` 信号把当前流误清掉的现象。

### 2.3 节点切换不会误杀 SOCKS

步骤：

1. 在 Shell 页启动 SOCKS bridge。
2. 记下状态栏显示的本地端口。
3. 保持同一节点，等待后台 snapshot 刷新。
4. 确认 SOCKS 仍保持监听。
5. 再切换到其他节点，确认桥接按预期关闭。

预期：

- 同一节点刷新时，SOCKS 不会被误停。
- 真正切换节点时，SOCKS 会被有意停止，不留下脏状态。
- 状态栏始终与实际监听状态一致。

### 2.4 下载不会因页面刷新中断

步骤：

1. 在 Shell 页填写下载路径并启动下载。
2. 下载过程中切换到 `Traffic`、`Collab`、`Loot` 等页面。
3. 手动触发一次聊天刷新和一次 Loot 刷新。
4. 等待下载完成。

预期：

- 下载进度持续更新，不因切页或刷新按钮而重置。
- 下载完成后状态标签正确显示 `completed`。
- 下载失败时能看到明确错误，不出现“按钮恢复了但线程还在跑”的状态错乱。

### 2.5 上传不会因页面刷新中断

步骤：

1. 在 Shell 页选择本地文件并启动上传。
2. 上传过程中来回切换页面。
3. 观察状态标签和开始按钮状态。

预期：

- 上传不会因切页、节点列表刷新或 Chat/Loot 刷新而中断。
- 进度标签持续更新。
- 上传结束后按钮状态恢复正确。

### 2.6 拓扑页频繁筛选不卡顿

步骤：

1. 打开 `Topology` 页。
2. 快速输入和删除筛选关键字至少 10 次。
3. 连续切换布局方式。
4. 反复开关 `show supplemental`。
5. 如有较大拓扑，执行一次 `Fit` 和一次 `Locate`。

预期：

- 输入和按钮响应不发黏，不出现明显 UI 假死。
- 视图刷新应有轻微延迟收敛，但不应每次按键都同步卡顿。
- 不出现边表与拓扑图高亮错位、选中状态乱跳。

### 2.7 Chat 刷新不乱跳

步骤：

1. 打开 `Collab` 页，发送一条消息。
2. 手动点击 `Refresh`。
3. 等待后台事件推送或再次刷新。
4. 检查列表顺序和去重行为。

预期：

- 已发送消息不会重复插入。
- 输入框在发送完成后被正确清空。
- 刷新过程中按钮会短暂禁用，但不会卡死或永久不可点击。

### 2.8 Loot 刷新不乱跳

步骤：

1. 打开 `Loot` 页，设置 `target / tags / category / limit`。
2. 连续点击 `Refresh` 两次。
3. 提交一个小文件作为 loot。
4. 再执行一次下载。

预期：

- 列表不会重复累加同一条 loot。
- 筛选条件不会在刷新后丢失。
- 提交和下载按钮在请求期间正确禁用，请求结束后正确恢复。

### 2.9 Listeners 刷新稳定

步骤：

1. 打开 `Listeners` 页，先刷新一次。
2. 创建、更新或删除一个 listener。
3. 在操作返回后再次刷新。

预期：

- 列表内容与实际状态一致。
- 失败时 toast 明确，成功时列表会更新，不出现“状态改了但表没变”。
- 不会因为多次快速点击导致按钮卡住。

---

## 3. 重点观察项

执行上面所有路径时，额外观察以下现象：

- 页面切换后焦点是否落在合理控件上。
- 按钮是否存在“永久 disabled”。
- 是否出现迟到的异步请求返回后覆盖当前状态。
- 是否出现同一条数据重复插入表格。
- 是否出现状态标签与真实运行状态不一致。

## 4. 建议记录方式

- 记录执行日期、环境和分支。
- 每项路径只记 `通过 / 失败 / 需复测`。
- 若失败，记录“步骤编号 + 现象 + 是否可复现”。
